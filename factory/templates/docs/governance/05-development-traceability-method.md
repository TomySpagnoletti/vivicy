# 05 - Development Traceability Method

Document status: stable development method.

## Purpose

This document defines how implementation must be driven from the architecture documentation without losing requirements.

The goal is not to make the documentation heavier. The goal is to turn the documentation into a traceable, executable development path:

```text
Current documentation
  -> frozen documentation baseline
  -> Requirement Catalog
  -> Traceability Matrix
  -> vertical issues with verification gates
  -> Architecture Graph issue index
  -> development progress overlay
  -> gate-first TDD implementation
  -> adaptive local and pre-production/integration verification gates
  -> traceability coverage report
  -> finished product
```

Implementation work must not start from a vague reading of the docs. Every development issue must point back to stable requirement IDs, and every requirement must eventually point forward to an issue, code, and a passing verification gate.

## Development Agent Operating Environment

The coding agent implementing this project is expected to work on a clean, dedicated development machine or workspace. This section applies to the development agent building {{PROJECT_NAME}}, not to the product's own runtime.

The development agent may install whatever it needs to implement and verify the project properly, including CLIs, system packages, language runtimes, package managers, container tools, cloud provider tools, database tools, browser automation tools, test utilities, and project-specific dependencies. There is no artificial tooling limit for the development mission.

The agent must not stop because a useful tool is missing if it can reasonably install that tool. Once an installed dependency becomes necessary for repeatable development, it must be captured in the appropriate setup script, package manifest, infrastructure script, or developer documentation instead of remaining an undocumented local assumption.

## Senior Autonomy Rule

The development agent building {{PROJECT_NAME}} must behave like a senior autonomous engineer, not like a junior executor waiting for step-by-step permission.

It must continue without owner interruption when it can resolve the issue by:

- reading the repository and canonical documentation;
- consulting official external documentation;
- installing local development tools;
- adding or updating tests and validation scripts;
- debugging failing gates;
- making reversible implementation choices that preserve the documented architecture;
- recording assumptions and evidence in traceability artifacts.

It may ask the owner only when forward progress requires an input the agent cannot create, infer, or verify:

- missing owner-provided credential;
- missing paid/provider account access;
- pre-production/integration account, IAM, billing, DNS, domain, or secret-store access that the agent cannot provision from existing credentials;
- external provider outage;
- contradictory source-of-truth documents;
- unresolved product or architecture decision;
- unavailable mandatory infrastructure;
- impossible verification gate.

The agent must not stop to ask about ordinary implementation details, file organization, package installation, test design, local debugging, or tool choice when the documented contracts and measured evidence are sufficient.

## Development Agent Repository Stewardship

The development agent building {{PROJECT_NAME}} is responsible for moving the repository forward autonomously.

It must manage:

- its own git worktree or branch strategy;
- logical commits after coherent green checkpoints;
- pushes when the remote is configured and credentials allow it;
- traceability updates before claiming an issue is done;
- validation evidence before each completion claim;
- preservation of unrelated local or user changes.

The development agent must not wait for the owner to ask for commits, pushes, or worktree hygiene after every implementation step. It owns that operational discipline.

This autonomy does not allow noisy or false checkpoints:

- no commit may be described as complete unless the issue's required gate is green;
- no WIP commit may be treated as a stable implementation checkpoint;
- no push may be used to hide a failing local state;
- no unrelated file may be reverted, reformatted, or swept into a commit for convenience;
- no product or architecture scope change may be folded into a commit unless an accepted Change Request has patched the canonical docs and produced a new verified frozen baseline after freeze.

## Freeze Before Extraction

Before extracting requirements, the documentation must be frozen for extraction.

Freeze does not mean the documentation can never change. It means:

- the owner has had a final review pass;
- obvious naming, scope, and architecture corrections are done;
- the current docs become a named, versioned, hashed baseline generated by the Doc Baseline Lock tool;
- requirement extraction is performed against that baseline;
- later doc changes create requirement changes instead of silently changing the build target.

The baseline lock process is defined in [Documentation Baseline Lock](08-doc-baseline-lock.md). The only repository-owned lock command is:

```text
node factory/doc-baseline.mjs generate --version <MAJOR.MINOR.PATCH> --status frozen --approved-by <approving-actor> --approval-ref <recorded-approval-reference> --out docs/baselines/<baseline_id>.json
node factory/doc-baseline.mjs verify --manifest docs/baselines/<baseline_id>.json --require-status frozen
```

A freeze requires a clean committed tree and recorded owner-approval evidence; the tool refuses otherwise ([Documentation Baseline Lock](08-doc-baseline-lock.md), Freeze Preconditions).

The active implementation baseline must be `status='frozen'` without a supersession marker. A `draft` manifest can be used only to test or review the locking mechanism.

If the owner changes the docs after extraction, the change must go through [Product Change Control](06-product-change-control.md), a new baseline must be generated when the CR changes canonical docs, the Requirement Catalog must be updated, and the traceability report must show what changed.

## Requirement Catalog

The Requirement Catalog is the canonical bridge between prose docs and implementation.

It must be authored and maintained as machine-readable and human-readable files at these canonical paths:

```text
spec/requirements/catalog.json
spec/requirements/catalog.md
```

The JSON file is the machine-readable source. The Markdown file is a human-readable report.

### Requirement Definition

A requirement is an atomic normative statement that can be implemented, verified, deferred, or explicitly classified as non-goal.

Not every line of documentation becomes a requirement. Some text is explanatory context, examples, rationale, future direction, or reading guidance. But every documentation line must be either covered by an issue requirement reference or deliberately excluded with a reason class so that nothing is silently ignored.

### Requirement ID Format

Use stable IDs:

```text
REQ-<AREA>-<NUMBER>
```

Examples:

```text
REQ-CORE-001
REQ-API-014
REQ-DATA-022
REQ-SECURITY-009
REQ-INFRA-004
```

IDs must not be reused after deletion. If a requirement is removed, mark it `removed` and explain why.

### Requirement Schema

Target shape:

```json
{
  "baselineId": "baseline-v1.0.0",
  "baselineVersion": "1.0.0",
```

## Coverage Model

The system must track three kinds of coverage.

### Source Coverage

Source coverage answers:

```text
Has every line of the frozen documentation been turned into executable work or deliberately excluded?
```

Source coverage is tracked at line level over exactly the files listed in the active frozen baseline manifest's `files[]`. Every line of every manifest file must be either covered by at least one issue requirement line reference or deliberately excluded with a reason class. Only three exclusions are mechanical and automatic: blank lines, pure code-fence delimiter lines, and horizontal rules. Every other excluded line requires an explicit exclusion record naming its reason class.

Each explicit exclusion must use one of these reason classes:

```text
supporting_context
example_scenario
rationale
future_option
evidence_needed
non_goal
reading_guidance
deprecated_or_removed
```

This prevents the common failure mode where an agent reads only the obvious feature sections and misses security, cost, memory, or infrastructure rules.

Section-level coverage is not sufficient. A section is covered only when every line inside it is either referenced by an issue requirement or explicitly excluded as context, example, rationale, future option, non-goal, evidence-needed, reading guidance, or deprecated.

### Implementation Coverage

Implementation coverage answers:

```text
Has every current-scope requirement been implemented and verified?
```

A current-scope requirement is complete only when it has:

- at least one linked issue;
- linked code or migration/config artifact when applicable;
- at least one passing verification gate;
- no open blocker;
- no unresolved ambiguity.

### Code Coverage

Code coverage answers:

```text
Have coverage-capable unit and integration test targets exercised governed application code at 100% lines, statements, branches, and functions?
```

The detailed code coverage gate, exclusions, and anti-cheating rules are defined in the Quality Gates section of this document.

## Spec-Driven Gate-First TDD

{{PROJECT_NAME}} uses a spec-driven method, but it does not adopt an external `spec.md -> plan.md -> tasks.md` pipeline as a source of truth.

The controlled source of truth remains the active frozen canonical documentation baseline. Accepted current-build Change Requests affect implementation only after they patch `docs/canonical/**/*.md` and produce a new frozen baseline.

```text
active frozen canonical documentation baseline manifest
  -> Requirement Catalog
  -> Source Map
  -> Traceability Matrix
  -> vertical implementation issues
  -> verification gates
```

No external method may create parallel primary specs, plans, tasks, constitutions, or backlogs alongside the canonical baseline and the traceability artifacts; those artifacts are the only specification chain.

Every implementation issue is gate-first:

1. Identify the requirement IDs and protected product truth.
2. Define the verification gate before implementation work is considered complete.
3. Add or update the required test, contract check, integration check, E2E scenario, integration smoke, or performance budget check as part of the issue.
4. When practical, prove the gate fails before the implementation.
5. Implement the smallest vertical slice that makes the gate green.
6. Refactor only while the gate remains green.
7. Update traceability before committing the completed checkpoint.

Tests are not a separate after-the-fact deliverable. They are the executable contract of the issue. A development sub-agent that implements a feature must also own the gate/test delta for that feature unless the master agent explicitly splits disjoint files for a non-overlapping support task.

## Architecture Graph Issue Index

Generated implementation issues must be linked to the architecture graph without mutating the frozen graph.

The target architecture map remains:

```text
docs/architecture-map/architecture-map.yml
```

`architecture-map.yml` is not included in the Doc Baseline Lock and must not become a live development status ledger after freeze. It indexes the canonical docs and is corrected when it conflicts with the active frozen canonical baseline.

The generated issue-to-graph bridge is:

```text
spec/development/issue-index.json
```

This file is authored by the semantic issue extraction pipeline's LLM write step from the Requirement Catalog, Traceability Matrix, and vertical issues. It is not product documentation and is not part of the frozen documentation baseline.

Each issue-index entry must include:

```text
id
title
summary
issue_path
requirement_ids
depends_on
spike_gates
graph_refs
verification_gate_ids
source_line_refs
```

`summary` is the issue's plain-language summary, surfaced on the architecture map so the owner can read what each issue does without opening the issue file.

Implementation order is first-class. `depends_on` lists the issue IDs that must be `verified` before this issue may start, and `spike_gates` lists the evidence spike gates that must be green before this issue may start. The issue index must record a valid topological order over `depends_on`: the `issues` array is listed in that order, and an index containing an unknown dependency ID or a dependency cycle is invalid. An issue may be claimed only when every issue in its `depends_on` is verified and all of its declared spike gates are green.

Top-level `issue-index.json` schema (this list is the schema the tooling enforces; `factory/generate-viewer-data.ts` is the executable validator):

```text
schema_version
status
baseline_id
baseline_version
manifest_path
manifest_hash
document_set_hash
source_corpus
verification_evidence_ref_grammar
coverage_summary
issues
```

Schema rules:

- `verification_evidence_ref_grammar` must equal the architecture map's `verification_gate_ref_grammar` exactly; the generator enforces this equality unconditionally, including while `issues` is empty. The map grammar is canonical; the issue index copy exists so `factory/progress-ledger.mjs` can validate evidence without parsing the map.
- The baseline identity fields (`baseline_id`, `baseline_version`, `manifest_path`, `manifest_hash`, `document_set_hash`) bind the artifact to the active frozen baseline. They are present even on the pre-extraction placeholder, and generation must fail when populated values do not match the active frozen manifest or when the pinned manifest is not `status='frozen'`.
- `coverage_summary.total_doc_lines` is a computed value, never a hand-seeded one: `semantic-extraction:check` computes it over exactly the active frozen manifest `files[]` using the non-blank rule defined in the Traceability Coverage Report section. The committed placeholder carries `0`, meaning "computed at extraction"; `semantic-extraction:check` must fail when the persisted value differs from the freshly computed value.
- There is no `graph_ref_resolution` field. Graph-ref resolution is enforced behaviorally (every `graph_ref` must resolve against the generated map, below), not stored as a schema field.

Issue-index entries do not store live progress status or completion evidence. The Traceability Matrix owns issue/gate completion state, and `progress-ledger.json` owns live local activity.

`graph_refs` must use only these forms:

```text
node:<node_id>
edge:<from_node_id>-><to_node_id>:<relation_slug>:<protocol_slug>
```

The architecture map generator emits the canonical `graph_ref` value for every node and edge. `issue-index.json` and `progress-ledger.json` must use those emitted values exactly. Every `graph_ref` must resolve to a node or edge generated from `docs/architecture-map/architecture-map.yml`. `semantic-extraction:check` or the architecture-map generator must fail when an issue references a missing graph item.

An implementation issue may reference several graph items when it cuts through a vertical product path. A graph item may be linked to several issues when it is implemented over several verified slices. Do not create graph-only issues; product implementation issues must still originate from active requirements.

## Development Progress Overlay

Live development-agent progress is recorded outside the frozen baseline:

```text
spec/development/progress-ledger.json
```

The progress ledger is the only local live progress source for the architecture map viewer.

It records:

```text
active_items
graph_item_states
issue_id
graph_refs
state
actor
worktree
session_ref
started_at
heartbeat_at
evidence_refs
```

The progress ledger also carries the same unconditional baseline identity fields as the issue index (`baseline_id`, `baseline_version`, `manifest_hash`, `document_set_hash`), so a ledger generated against a stale baseline is detectable.

A graph item may be linked to several issues, so each `graph_item_states` entry records a per-issue status map (`issue_states`) and a single displayed `status` computed as the conservative aggregate: the displayed status is the least-advanced status across the item's linked issues, and it must not become `verified` while any linked issue's recorded state remains unverified, blocked, or in progress. One issue going green never overstates a shared node or edge.

The target graph stays stable. The viewer overlays `issue-index.json` and `progress-ledger.json` on top of the graph.

Progress semantics:

- a slow blinking green dot means the development agent has an active heartbeat for that issue or graph item;
- `in_progress` means implementation or review is active but the issue is not green;
- `implemented` means code exists locally but required verification is not complete;
- `verified` means the required issue gate passed and evidence is recorded;
- `blocked` means a real stop condition is recorded with evidence.

`verified` must not be written without a green verification gate reference. `implemented` must not be used to bypass gates. Activity heartbeats are live telemetry, not proof of completion.

Green verification gate evidence has a machine definition. It is a gate-run record: a JSON file under the verification evidence grammar (`spec/development/gates/...`) with at least:

```text
gate_id
issue_id
command
exit_code
status: pass | fail
finished_at
baseline_id
```

The gates directory is the gate registry: the `verification_gate_ids` declared on an issue are not free-text labels; they must be the `gate_id` values of gate-run records produced by actually running the gate.

Binding rule for `verified`: a `gate_passed` or `issue_completed` event may flip a graph item to `verified` only when at least one of its `evidence_refs` resolves to a gate-run record with `status: pass` and `exit_code: 0` whose `gate_id` is one of the linked issue's declared `verification_gate_ids`. `issue_completed` asserts that the issue's own verification gate has passed; it carries exactly the same gate-evidence obligation as `gate_passed`, never less. A path that merely matches the grammar, or a passing record for some other issue's gate, is not verification evidence for this issue. `factory/progress-ledger.mjs` enforces this binding at write time for issues that declare `verification_gate_ids`; `semantic-extraction:check` and `traceability:check` must enforce the same rule over generated artifacts.

Every progress-ledger `evidence_refs` entry must be a repository-relative path or path plus line that resolves on the current filesystem before viewer data is generated.

Evidence matrix:

```text
heartbeat-only active item -> heartbeat_at required, evidence_refs optional
in_progress graph item -> issue_id and heartbeat or issue claim required, evidence_refs optional
implemented graph item -> implementation evidence_refs required
verified graph item -> green verification gate evidence_refs required
blocked graph item -> blocker evidence_refs required
```

The viewer may show generated issues through a Tasks panel, but it must not control the development agent. It is a read-only observation surface for owner understanding and agent coordination.

## Local Development Progress MCP

Local development-agent progress uses one reporting protocol:

```text
local development progress MCP
```

The graph viewer does not talk to the development agent directly and does not issue work commands. The development agent, or the master/spec-steward session coordinating its work, reports progress events through the local development progress MCP (`factory/progress-mcp.mjs`).

The MCP exposes one canonical write tool: `development_progress.record_event`. The event types below are values of that tool's `event_type` field, not separate tools. The documented taxonomy and the implemented tool surface must not diverge silently: changing either requires updating the other.

Minimum event types:

```text
issue_claimed
issue_started
graph_item_focus
heartbeat
gate_started
gate_passed
gate_failed
issue_blocked
issue_completed
issue_reopened
review_started
review_completed
```

`issue_reopened` is the only event allowed to move a graph item out of a terminal status (`verified`/`implemented`) back to `in_progress`; it models a deliberate retraction of verification, never an accidental downgrade from a late or out-of-order event.

Every event must include:

```text
issue_id
graph_refs (non-empty: the event's explicit graph focus)
actor
session_ref
worktree, when applicable
timestamp (server-assigned by the MCP at record time when omitted)
evidence_refs, when applicable
```

`state` is not a caller-supplied event field: the ledger derives it deterministically from `event_type`, so the wire contract cannot claim a state its event type does not produce. The MCP input schema must reject events missing `issue_id`, non-empty `graph_refs`, `actor`, or `session_ref`; an empty `graph_refs` fallback to "all of the issue's graph refs" is not allowed.

The MCP server initializes and owns writes to `spec/development/progress-ledger.json`. The development agent emits progress events through the MCP; it must not edit the ledger directly. Other scripts may read it. The architecture map generator validates and embeds it into viewer data. Do not add a second live progress protocol, ad hoc HTTP writer, or manual status file.

When a development agent CLI exposes a read-only session-context surface, it can be used as a read-only source of local session context when available, but it is not the progress write protocol. Any useful observation must be normalized into the same progress ledger.

## Implementation Issue Prompt Shape

Every implementation issue is a vertical slice and an execution prompt. It must follow the intent-first order defined in this document.

Issues are authored by LLM agents — the orchestrator and its sub-agents — reading the frozen canonical baseline directly, using the repository issue template at `spec/development/ISSUE-TEMPLATE.md`. Authoring is a lossless transformation of the frozen docs, not a derivation: every issue carries a plain-language Summary, surfaced on the architecture map through the issue index, and explicit requirement line references into the frozen docs.

The issue must not exist without at least one requirement ID unless it is a pure refactor, dependency update, or exploratory spike. Spikes must reference `evidence_spike` requirements.

Every issue must contain these sections in this exact order:

```text
Context
Primary Objective
Why This Change Exists
Protected Product Truth
Non-Negotiable Constraints
Protected Product And UI/UX Constraints
Spec Target
Required Implementation Outcomes
Anti-Cheating Rules
Definition Of Done
Validation To Run
Final Response Format
Stop Conditions
Traceability Update
```

### Task Type And Writing Discipline

Every issue declares a task type that sets type-specific behavior the issue must encode:

- `implementation`: deliver the behavior together with its tests (gate-first TDD).
- `diagnosis`: reproduce, localize, and explain the cause; do not silently expand into a fix beyond the stated scope.
- `cleanup` or `rewrite`: state the target simplification pattern explicitly — the single authority or source of truth that must remain, and the named complexity to remove — and the exact user-visible behavior and contracts that must be preserved while the implementation shape changes.
- `review_fix`: report findings first; apply only bounded, intent-preserving fixes within scope, never silent out-of-scope mutations.

Writing discipline for every issue: use deterministic wording. Do not use discretionary phrasing — "if needed", "if possible", "if justified", "simplify where appropriate", "clean up this area", "reduce complexity", "fix the tests", "refactor this flow". State instead the exact target state, the owners, the behavior to preserve, and the gates. Intent and protected truth come before implementation mechanics, which the section order above already enforces.

### Context

Required fields:

```text
Current repo state summary
Relevant architecture documents
Relevant existing modules, schemas, scripts, or tests
Declared canonical truth source for the issue
Relevant execution context sources
```

The declared canonical truth source must be:

```text
frozen_doc_baseline
```

Execution context sources may include `current_filesystem_state`, `executable_contract`, or `spike_result`, but they must not become product truth outside the canonical baseline. After documentation freeze, an owner decision that changes product behavior, architecture, active scope, UX, data model, protocols, security, cost, or mandated implementation order must patch `docs/canonical/**/*.md` through Product Change Control and produce a new frozen baseline before issue extraction uses it.

### Issue Granularity

One issue solves one problem. Prefer many small, single-responsibility issues over a few large ones: the narrower the scope, the easier the issue is to understand, implement, test, review, and verify.

The lower bound is an independently testable vertical slice. An issue must be as small as possible but no smaller than one coherent, observable behavior that carries its own verification gate. Do not split below that bound: a fragment with no behavior testable on its own — for example a schema column or a helper with no exercised path — is not a valid issue and must be folded into the slice it serves.

The upper bound is single responsibility. If an issue bundles outcomes that could each ship and be verified separately, split it. The Primary Objective states one outcome — it may be a single coherent flow, never a list of independent ones.

### Primary Objective

State the single end-to-end outcome the issue must produce. The objective must describe observable product or platform behavior, not only files to edit.

Good:

```text
An authorized inbound request is validated, persists its state transition, and returns one normalized response through the public interface.
```

Bad:

```text
Create the request module.
```

### Why This Change Exists

State the product or engineering reason before implementation mechanics. This section protects intent from scope drift.

### Protected Product Truth

State the invariant or user-facing contract that must remain true after the issue. Draw it from the canonical baseline. Examples (your product defines its own):

```text
Subsystem X reaches platform state only through its single defined protocol.
The privileged component never receives the protected secret value.
Cost data is recorded for review but is not default agent context.
```

### Non-Negotiable Constraints

Required fields:

```text
In-scope requirements
Out-of-scope requirements
Relevant non-goals
Relevant security, memory, runtime, or cost invariants
Forbidden fallback paths
```

### Protected Product And UI/UX Constraints

Every issue must explicitly state whether it touches authorized-human-facing UX. If the issue is not a UX task, it must forbid incidental UI or copy changes except for text required by the issue.

### Spec Target

Required fields:

```text
Requirement IDs

Baseline version

Baseline document set hash
Requirement line references into the frozen docs (`path:startLine-endLine`)
Source excerpt hashes
Depends-on issue IDs

Requirement maturity
Requirement disposition
```

### Required Implementation Outcomes

List exact deliverables. A deliverable can be code, schema, migration, infrastructure module, image recipe, contract test, integration test, E2E gate, or documentation update required to keep docs truthful.

Layer-only tasks are not valid unless the issue is explicitly classified as an infrastructure or contract slice with its own verification gate.

Each issue must state the required test and gate delta before implementation starts:

```text
New tests or gates
Updated tests or gates
Existing gates that prove no regression
Performance budget checks touched
Pre-production / integration gates required before release readiness
```

### Anti-Cheating Rules

Each issue must forbid the shortcuts most likely to produce a false green result.

Mandatory defaults:

```text
Do not weaken assertions to pass tests.
Do not skip, narrow, or mark tests pending to pass the gate.
Do not replace a real boundary with an in-memory shortcut unless the issue explicitly allows a fake at the selected verification level.
Do not introduce a second protocol, fallback path, duplicate source of truth, or shadow state store.
Do not claim validation from commands that were not run on the current filesystem state.
Do not relax performance budgets, narrow assertions, skip covered scenarios, add permissive retries, or downgrade the verification level to make the gate pass.
Do not use mock-only proof for stateful mutations, provider integrations, lifecycle behavior, billing/cost behavior, secrets behavior, or persistent-state behavior that requires real state.
Do not split tests and implementation into two competing interpretations of the requirement.
```

Post-freeze product ideas must not be folded into an implementation issue directly. They must first pass through [Product Change Control](06-product-change-control.md). Only an accepted current-build Change Request can update active issue scope, traceability rows, or verification gates.

### Definition Of Done

Required fields:

```text
Exact behavior that exists
Exact behavior that is forbidden
Exact state/schema/protocol condition that is true
Exact requirement IDs covered
Exact verification gate status
Exact local and pre-production evidence required for the issue's maturity
Residual risks, only if real
```

An issue is not done until its verification gate is green.

### Validation To Run

Every issue must define how it becomes green.

Required fields:

```text
Gate ID
Baseline ID
Baseline version
Baseline manifest path
Manifest hash
Document set hash
Verification level
Environment
Command or automated check
Expected observable result
Mocks/fakes allowed
Real components required
Pre-production environment requirement, if any
Performance budget class and threshold source, if any
Requirement IDs verified
Failure handling
```

Success may be claimed only for validation commands actually run against the current filesystem state.

### Final Response Format

The implementation agent must report:

```text
Short summary
Requirements covered
Files changed
Validation commands and results
Pre-production evidence, if required
Performance budget evidence, if touched
Traceability updates
Commit and push references, if available
Residual risks only if real
```

### Stop Conditions

The implementation agent stops only when:

```text
All required implementation outcomes are complete.
All validation commands listed in the issue pass.
The traceability matrix is updated.
The coherent green checkpoint is committed and pushed when the remote and credentials are available.
```

The agent may stop early only for a real blocker:

```text
missing owner-provided credential
missing paid/provider account access
missing pre-production/integration account, IAM, billing, DNS, domain, or secret-store access
external provider outage
contradictory source-of-truth documents
unresolved product decision
unavailable mandatory infrastructure
impossible verification gate
```

Partial progress summaries are not completion.

### Traceability Update

At completion, the issue must update both the Requirement Catalog and the Traceability Matrix status for every covered requirement when either artifact changes. The issue cannot be marked complete with one stale artifact.

## Adaptive E2E Ladder

The system is too large to require full real infrastructure from the first issue. Verification must become more realistic as the system matures.

Use this ladder (rename the higher rungs to your runtime; the principle is fixed):

```text
L0 - Static contract tests
L1 - In-process integration with fake adapters
L2 - App wired to a local real datastore (e.g. local database)
L3 - Real external adapter in test mode plus fake downstream component
L4 - Real supervisor/host process on a local runtime simulation
L5 - Local runtime simulation with persisted state
L6 - Pre-production deployment using the same infrastructure modules, service topology, managed services, provider integrations, and runtime boundaries as production, with separate pre-production data, secrets, records, domains or delegated test domains, external identities, and cost tags
```

Each issue chooses the highest currently available verification level allowed by the promotion rules below. A lower-level gate is allowed only when the required higher-level runtime does not exist yet; the issue must name the missing runtime, the fake or simulation being used, and the exact later gate that must replace or promote that proof.

The Traceability Matrix must show when a requirement is only covered by a lower-level gate and still needs a higher-level E2E gate.

### Pre-Production Rule

Local gates are necessary but not sufficient for release-quality proof.

The project must have a pre-production environment that is intentionally production-like:

- same infrastructure module family as production;
- same service shape as production;
- same runtime/host implementation as production;
- same database/migration path as production;
- same external integration paths as production, with pre-production identities;
- same managed-service classes as production where the feature touches them;
- separate pre-production credentials, secrets, data, domains, external identities, cost tags, and records.

A feature that changes deployable runtime behavior, infrastructure, provider integration, isolation boundaries, secrets, or persistent state is not release-ready until its required L6 pre-production gate passes.

The development agent must be able to deploy, smoke test, diagnose, and tear down or preserve this pre-production environment through documented commands. The pre-production gate must be automated enough that another agent can rerun it without relying on local memory.

The traceability matrix must distinguish these verification stages separately from implementation status (this is the defined vocabulary of the matrix's "Verification stage" column):

```text
locally_done        -> the issue's local gate is green on the current filesystem
locally_integrated  -> proven against real integrated runtimes/dependencies, not just unit-level fakes
preprod_verified    -> proven in the production-like pre-production environment
release_ready       -> all required verification levels for the requirement's area are met
```

No release-scoped requirement can be `release_ready` without the highest verification level required by its area.

## Quality Gates And Performance Budgets

The first implementation must create root quality gates as executable package scripts or equivalent commands. This document defines the doctrine; once scripts exist, the scripts and tests become the executable source of truth.

A pre-implementation repo-level surface already exists: the root workspace exposes reproducible install, `npm test`, the dependency audit surface (for the Dependency Audit Gate below), and `npm run verify` (active-baseline verification, progress-ledger tests, architecture-map generator and viewer typechecks). Product root gates must extend this root script surface rather than adding parallel per-package entry points.

Root gates must cover, at minimum:

```text
lint
typecheck
build
unit tests
integration tests
code coverage checks for unit and integration test targets
database/migration checks
contract/schema checks
governance/drift checks
dependency vulnerability/freshness audit (waiver-based, non-blocking; see Dependency Audit Gate)
i18n governance checks (when the product is localized)
E2E gates for governed surfaces
pre-production smoke gates when required by the issue
```

Root gates are confirmation gates. They are not the first place to discover predictable missing test classes. The implementation issue must identify and add the required test or gate delta before the root gates are launched.

### Autonomous Development Workflow

Development is performed autonomously by an AI development agent and its sub-agents. There is no human implementation team and no human code review:

- the agent manages its own branches, worktrees, commits, and pushes, as defined in Development Agent Repository Stewardship;
- there are no pull requests and no review queue: the agent integrates its own branches and worktrees once the required gates are green;
- the correctness guarantee is the gate system, not review ceremony: the root `npm run verify` surface (extended by the product root gates), the per-issue verification gates, the production-like tests defined in each issue, and the required pre-production checks must all be green before any advance;
- the agent escalates to the owner only on real blockers or product decisions, never for routine workflow approval;
- the pre-production deploy/smoke gate (L6) runs through the documented commands with scoped credentials, in a CI-or-operator-workstation execution context, so another agent can rerun it without local memory.

A server-side CI runner is explicitly deferred: it becomes useful as a neutral arbiter on `main` only when parallel multi-machine development starts, and it must then run the same executable gates — never a parallel gate definition. Until then, the local-first rule stands — no push may hide a failing local state, and a red gate is a broken gate to fix before new feature work.

### Governance Drift Check

The `governance/drift checks` root gate is mechanical, not a prose review. Its minimum content is:

1. Baseline integrity: `node factory/doc-baseline.mjs verify --manifest <active frozen manifest> --require-status frozen` (with the pinned `--require-baseline-id`). This is the only mechanical alignment proof for the canonical doc set itself.
2. Reference validity: every `docs/**/*.md` path referenced by `AGENTS.md`, `README.md`, and `docs/governance/**/*.md` must exist on disk. A single command is sufficient, for example:

```bash
grep -rhoE '\bdocs/[A-Za-z0-9._/-]+\.md\b' AGENTS.md README.md docs/governance | sort -u | while read -r p; do [ -e "$p" ] || { echo "MISSING: $p"; exit 1; }; done
```

Beyond these two checks, the prose summaries in `AGENTS.md` and `README.md` are human-reviewed convenience text and are not mechanically proven aligned with canonical docs. Their correctness claim is exactly that: human-reviewed, with the canonical-wins conflict rule as the safety net. Do not present them as verified, and do not build a heavier semantic alignment checker for them.

### Dependency Audit Gate

The `dependency-audit` gate scans the locked dependency tree for known vulnerabilities (for example `npm audit` against an advisory database) and for abandoned or outdated packages. It is a triage gate, not a zero-vulnerability gate, and must never block a release on its own: it fails only on a high or critical advisory that is not in a recorded exceptions allowlist. Each exception records the advisory or package, the reason no fix is available, the accepted risk, and a review-by date. Known-unavoidable obsolete or vulnerable packages therefore pass through a documented exception rather than blocking production. Dependency review and updates run on a monthly maintenance cadence that refreshes or expires exceptions and applies available patches; a weekly cadence is not required.

### External-Interface Diagnostic Evidence

When an issue implements or changes an external protocol server, client driver, namespace, or tool/message schema, the development agent may use an interactive inspector to diagnose the live handshake, capability listing, schemas, sample calls, transport behavior, and protocol errors.

Inspector output is supporting diagnostic evidence. It is not a release gate and does not replace automated contract, integration, E2E, or pre-production tests. If the inspector exposes behavior that protects a requirement, the issue must encode that behavior in an executable gate before it can be marked complete.

Inspector sessions must use local or private pre-production endpoints and dedicated, least-privilege, short-lived, session-scoped test credentials for synthetic local or pre-production records only. They must not require production tokens, expose public endpoints, use shared public tunnels, or expose secret values, raw provider payloads, raw transcripts, raw files, or raw command logs.

Inspector captures and representative call samples must use synthetic minimal fixtures and must be redacted before storage or sharing. Store only schemas, method names, status/error codes, trace IDs, byte counts, and bounded sanitized excerpts when explicitly needed.

### Code Coverage Gate

Code coverage is a hard implementation gate. It is separate from source coverage and implementation coverage:

- source coverage proves that frozen documentation lines were covered by issues or deliberately excluded;
- implementation coverage proves that requirements are linked to issues, artifacts, and verification gates;
- code coverage proves that executable tests exercise the application code they claim to protect.

Every package, service, library, or script whose product or runtime logic is protected by unit or integration tests must use coverage-capable tooling and enforce 100% coverage for all four counters:

```text
lines: 100%
statements: 100%
branches: 100%
functions: 100%
```

At issue level, this threshold applies to all governed application code created or modified by the issue before the issue can be marked `locally_done`. At root-gate or release-gate level, it applies to every governed coverage-capable unit and integration test target in the repository or workspace. A package with uncovered governed application code cannot pass by narrowing the issue-local coverage command.

Allowed exclusions are limited to:

- generated files;
- vendored third-party code;
- static config or manifest files with no executable branch;
- framework bootstrap entrypoints and index barrels with no decision logic;
- type-only declarations;
- test fixtures and test factories;
- mechanical migration files whose behavior is verified by database or migration gates.

Each exclusion must be explicit in the coverage configuration, use a stable path pattern, and include a reason. Exclusions must not hide business logic, authorization, isolation helpers, security checks, protocol adapters, schema validation, state transitions, orchestration, lifecycle behavior, persistence behavior, cost logic, i18n behavior, or error handling.

Do not move logic into generated, config, bootstrap, barrel, fixture, or migration files to avoid coverage. If a file contains decisions, branching, side effects, parsing, authorization, persistence, provider calls, protocol handling, orchestration, or error behavior, it is governed application code.

Coverage cannot be satisfied by empty tests, shallow import tests, snapshot-only tests, or tests that execute lines without asserting behavior. The 100% threshold is necessary but not sufficient: tests must assert behavior, boundaries, authorization, errors, and state transitions tied to requirement IDs when those concerns exist. If a test runner cannot produce one of the four required counters for a target that contains governed application logic, the implementation must add or configure coverage tooling before that target can be completed.

I18n governance checks (when the product is localized) must fail when implementation introduces:

- hardcoded product-facing strings in runtime UI;
- hardcoded human-visible backend/system messages outside i18n resources;
- missing translation keys for governed user-facing surfaces;
- an i18n package outside the canonical stack;
- locale resolution that does not follow the implementation contract.

Repo-owned validation scripts must fail with corrective output that includes:

```text
Rule
Scope
Evidence
Expected
Required fix
```

### Governed E2E Surfaces

{{PROJECT_NAME}} must not attempt to govern every surface with full E2E weight from the first commit.

The first implementation must define a small governed E2E set for critical paths. The exact governed set is derived from requirements and implementation maturity. Once a surface is governed:

- every governed scenario must have a stable scenario ID;
- the canonical E2E gate must verify that required scenarios actually ran;
- unexpected runtime errors, failed requests, and server 5xx responses must fail by default unless a scenario-local allowlist proves the failure is intentional;
- governed coverage must remain mechanically complete for the declared governed set.

### Performance Budgets

Performance budgets are product and runtime contracts, not nice-to-have metrics.

Budget classes must be tied to requirement IDs and verification gates. The project owner defines budget classes from its own critical paths (request latency, end-to-end roundtrips, cold-start readiness, deploy smoke, and so on).

The first exact thresholds must come from this project's evidence, early implementation baselines, or an accepted requirement. A threshold must not be copied from another project without this project's evidence.

When a budget fails, the required response order is:

1. reproduce the failure;
2. measure with the authoritative gate;
3. localize the slow path;
4. distinguish product latency from environment, fixture, provider, cold-start, or contention effects;
5. apply the smallest evidence-backed fix;
6. rerun the governed measurement.

The implementation agent must not relax a budget, replace a real measurement with a synthetic harness, broaden scope into a speculative rewrite, or hide the failure behind retries without explicit owner-approved requirement change.

## Development Loop

An issue may be claimed only when every issue in its `depends_on` list is verified and all of its declared evidence spike gates are green. The topological order recorded in the issue index is the default execution order.

For each issue:

1. Emit an `issue_claimed` event through the local development progress MCP so the MCP records the claim in `progress-ledger.json`.
2. Read the linked requirements.
3. Read referenced source doc sections.
4. Confirm the public behavior and verification gate.
5. Emit active graph refs through the local development progress MCP.
6. Write the failing test or executable check first when practical.
7. Implement the smallest vertical slice that can pass.
8. Run the gate.
9. Fix until green.
10. Refactor only while green.
11. Update the Requirement Catalog and Traceability Matrix as required; regenerate the issue index through the semantic issue extraction pipeline; emit progress events so the MCP updates the progress ledger.
12. Commit the coherent green checkpoint.
13. Push when the remote and credentials are available.
14. Move to the next issue only after the current gate is green and the traceability update is recorded.

This follows a red-green-refactor loop, but at the issue level:

```text
spec target
  -> failing gate
  -> implementation
  -> green gate
  -> traceability update
  -> progress overlay update
  -> commit/push checkpoint
```

The master/spec-steward session may run several local development-agent conversations and worktrees in parallel when issues are independent. Parallel work is allowed only when each active issue has:

- a distinct issue claim;
- explicit `graph_refs`;
- a dedicated worktree or non-overlapping workspace when files can conflict;
- a visible progress heartbeat;
- the standard spec review and code/gate quality review before completion;
- a green gate before the issue can be marked complete.

Do not use parallelism to stack unverified work. Parallelism is for independent, traceable, gate-first slices.

## Development Agent Orchestration

The recommended development pattern is not parallel implementation and parallel test writing for the same feature. That creates competing interpretations of the same requirement.

Use this model instead:

```text
Master / Spec Steward
  -> Gate-First Implementer
  -> Spec Reviewer
  -> Code And Gate Quality Reviewer
  -> Reconciler only when a real conflict exists
```

### Master / Spec Steward

The master agent owns traceability and completion judgment. It prepares the vertical issue from the Requirement Catalog and Traceability Matrix, provides only the required context to sub-agents, checks the diff and validation evidence, and decides whether the issue is actually done.

The master agent should not code except for small integration fixes or documentation wiring. Its job is to protect the source of truth.

### Gate-First Implementer

The implementer owns the feature and its executable proof together.

It must:

- read the issue's requirement IDs and source excerpts;
- define or update the verification gate;
- add or update the test delta before relying on root gates;
- implement the vertical slice;
- run the required gates;
- update traceability;
- commit the coherent green checkpoint when allowed.

### Reviewers

Run reviews in order:

1. Spec reviewer: checks requirement coverage, source truth, scope, non-goals, forbidden fallbacks, and absence of extra behavior.
2. Code and gate quality reviewer: checks architecture, maintainability, test strength, anti-cheating, performance-budget handling, and validation evidence.

If a reviewer finds an issue, the implementer fixes it and the relevant review repeats. A self-review by the implementer does not replace these reviews for normal issues.

### Reconciler

Use a reconciler only when there is a real conflict:

- implementation and gate disagree about the requirement;
- docs and executable contract disagree;
- the required verification level is disputed;
- a proposed solution introduces a second source of truth, second protocol, fallback path, or shadow state;
- the issue appears to require a product or architecture change.

The reconciler does not average opinions. It resolves product conflicts against the active frozen canonical baseline. Executable contracts, spike evidence, and current filesystem state can prove implementation behavior or evidence status, but they do not change product truth unless the canonical docs are patched and a new frozen baseline is verified.

### Autonomous Two-Agent Loop

These roles may run as an external, deterministic orchestrator (`factory/dev-loop.mjs`) that drives separate agent CLIs per issue rather than one self-looping agent. The Gate-First Implementer and the review roles are DISTINCT agents — each running its own review sub-agents — so the reviewer is never the issue's author. The specific CLI assigned to each role (for example Claude Code or Codex) is configurable in the loop. The review agent applies bounded, intent-preserving fixes and re-runs the gate; the orchestrator then re-runs the gate itself as the authoritative verdict (never an agent's claim), commits the green checkpoint, and moves the issue file into `spec/development/issues/done/`.

One issue is one conversation: each agent leg is a fresh CLI invocation with no carryover. The issue contract plus its referenced canonical lines is the complete context; durable state lives in the repo, canonical docs, the issue index, and the ledger — never in agent memory. Anything a later issue needs is read from the code, the canonical baseline, or the ledger; a defect discovered in the baseline goes through Product Change Control, not agent memory.

The loop runs a bounded number of cycles (implement → review & fix → gate) per issue; on a gate still red after the bound it records `issue_blocked` with evidence and stops for a human. It is sequential by default and resumes from the ledger and the `done/` folder.

Progress reporting is hook-driven for determinism rather than trusting an agent to remember. Lifecycle hooks emit the mechanical events (`issue_started`, `review_started`, `heartbeat`, gate results) and inject each agent's `actor` and `role`; the agent emits only the semantic events a hook cannot judge (`review_completed` with an iso/not_iso verdict, `issue_blocked` reason, `graph_item_focus`); a Stop hook emits a fallback when an agent finished a leg without reporting. Hooks and the agent both emit through the same validated path (the progress MCP tool, or `factory/progress-emit.mjs` for hooks), never hand-editing the ledger. Events carry `role` so the architecture map shows which agent is implementing versus reviewing, with review as a distinct `reviewing` state.

## Implementation Workflow

Apply this method directly: challenge unclear requirements against the canonical docs and domain language, break requirement groups into vertical implementation issues, implement each issue gate-first with behavior tests, and diagnose any gate that fails. Requirement and issue generation is bound exclusively to the canonical semantic issue extraction process defined below — the `semantic-extraction:write` LLM extraction step controlled by the deterministic `semantic-extraction:check` and `traceability:check` commands and the LLM fidelity review — and no parallel path may become a second extraction authority.

The first development prompt for launching this work is maintained in [Development Launch Prompt](07-development-launch-prompt.md). It is a starting prompt for the coding agent, not a second source of truth. If it conflicts with this method, this method wins.

## Semantic Issue Extraction Process

Semantic extraction is LLM work controlled by a deterministic check and an independent fidelity review. The pipeline has one canonical write step and two control layers:

```text
semantic-extraction:write
  -> the LLM extraction step: the orchestrator and its sub-agents read the frozen baseline and author the Requirement Catalog, Traceability Matrix, vertical issues, and issue index

semantic-extraction:check
  -> npm run semantic-extraction:check (script: factory/semantic-extraction-check.mjs): deterministic verification of line coverage, reference resolution, and structural invention; produces the Source Map and the traceability coverage report

traceability:check
  -> fail when current-scope requirements lack issues, gates, test/code references when applicable, or verified status after completion
```

There must be one canonical write path and one canonical check path. `semantic-extraction:write` is not a deterministic code command: it is the LLM extraction step, performed by the orchestrator and its sub-agents against the frozen baseline. The deterministic commands are `semantic-extraction:check` and `traceability:check`. Agents must not maintain several extraction paths or ad hoc prompt-only mappings.

### LLM Extraction Step (Write Side)

The write side is reading and authoring, not parsing. The orchestrator (the master/spec-steward session) and its sub-agents must:

1. Read only canonical product documentation files included by the active frozen Canonical Documentation Baseline: `docs/canonical/**/*.md`, exactly as listed in the manifest's `files[]`.
2. Author vertical implementation issues using the repository issue template at `spec/development/ISSUE-TEMPLATE.md`.
3. Perform a lossless transformation, not a derivation: the issues restate what the frozen docs require, with no invention, no compression of normative content, and no silent omission.
4. Give every issue a plain-language Summary, surfaced on the architecture map through the issue index.
5. Give every issue explicit requirement line references (`path:startLine-endLine`) into the frozen docs.
6. Record every deliberately excluded doc line with its exclusion reason class.
7. Assign or preserve stable requirement IDs.
8. Declare `depends_on` issue IDs and evidence spike gates on every issue, and keep the issue index in a valid topological order.
9. Produce the Requirement Catalog.
10. Produce the Traceability Matrix.
11. Produce the Architecture Graph issue index.

### Deterministic Control (`semantic-extraction:check`)

`npm run semantic-extraction:check` (script: `factory/semantic-extraction-check.mjs`) mechanically verifies the LLM output without interpreting meaning:

- Line coverage: every line of every file in the frozen manifest `files[]` is either covered by at least one issue requirement line reference or deliberately excluded with a reason class. Blank lines, pure code-fence delimiter lines, and horizontal rules are auto-excluded mechanically; every other excluded line requires an explicit exclusion record.
- Reference resolution: every issue requirement reference resolves inside the frozen baseline — the referenced file must be in the manifest and the line range must be valid. Unresolvable or empty references fail the check.
- Structural invention: an issue claiming documentation content must point at real lines. Claimed content with no resolvable source lines is caught as invention.

The check produces the Source Map and the traceability coverage report under `spec/requirements/`. It cannot judge whether prose was transformed faithfully; that is the fidelity review's job.

### LLM Fidelity Review (Control Part 2)

Deterministic checks prove coverage and resolution, not meaning. A fleet of independent review sub-agents — none of which authored the issue under review — reviews each issue against exactly the doc lines it references and returns one verdict:

```text
iso     -> the issue restates the referenced lines with no added meaning, no lost meaning, and no drift
not_iso -> the issue adds, loses, or shifts meaning relative to the referenced lines
```

This is a natural-language fidelity judgment, the part a deterministic script cannot make.

Verdicts flow back to the orchestrator as structured outputs. The orchestrator is the single writer: it compiles the verdicts and persists the verdict report atomically (temp-file write plus rename, the same guarantee as the progress ledger) under `spec/development/reports/`. Review sub-agents never write artifacts; no concurrent writers exist by construction.

Every `not_iso` issue must be corrected and re-reviewed until its latest verdict is `iso`. `semantic-extraction:check` must re-run after every correction, because a correction can change line coverage or references.

### Change Request Handling

Change Request handling is status-aware:

- `docs/change-requests/CR-TEMPLATE.md` is classified as template/governance and must not generate implementation requirements.
- Actual CR instance files under `docs/change-requests/CR-*.md` are excluded from baseline source extraction. They are audit and delta records, not active requirement sources.
- A CR with status `accepted_current_build` or `docs_applied` can identify impacted canonical docs, impacted requirements, and expected catalog deltas, but it must not be the sole source of an active requirement. The canonical docs must be patched and a new frozen baseline must be generated (the CR is then `docs_applied`) before the active Requirement Catalog treats the change as execution truth.
- CRs with status `accepted_future` are future options and must not generate current-build issues.
- CRs with status `idea`, `under_review`, `rejected`, `superseded`, or any unknown status are registry/context/non-goal material for the active build and must not generate current-build issues.
- If a generated current-build requirement is sourced only from a CR instance file, `semantic-extraction:check` must fail.
- `semantic-extraction:check` must also run the CR Registry Validation rules defined in [Product Change Control](06-product-change-control.md) and fail on any invalid CR file.

`semantic-extraction:check` must produce:

```text
spec/requirements/source-map.json
spec/requirements/source-map.md
```

`source-map.json` is the mechanical proof that every manifest line is either covered by one or more issue requirement references or excluded. A line with zero requirement references is allowed only when a mechanical auto-exclusion applies or an explicit exclusion record explains why it is not executable.

The write step must also produce:

```text
spec/development/issue-index.json
```

`issue-index.json` is the mechanical bridge between generated issues and the architecture map. It must include the raw line coverage summary used by the viewer Tasks panel and the coverage report. It must not invent graph refs; every graph ref must resolve against `docs/architecture-map/architecture-map.yml`.

`semantic-extraction:check` must also produce the Traceability Coverage Report at:

```text
spec/requirements/coverage-report.json
spec/requirements/coverage-report.md
```

The process must be stable. Re-running extraction without doc changes must not create new IDs or reorder unrelated requirements, and running `semantic-extraction:check` twice over unchanged artifacts must produce the same result.

Artifact existence is gated. The semantic issue extraction pipeline is itself the first implementation issue; until it runs, `spec/requirements/` does not exist and the development-control files under `spec/development/` are explicit placeholders (`issues: []`, zero coverage counters). Once extraction has run, a missing Source Map, Requirement Catalog, Traceability Matrix, issue index, or coverage report is a hard `semantic-extraction:check`/`traceability:check` failure, and no implementation issue beyond the pipeline itself may start while a required artifact is missing.

### Source Reference Contract

The extraction write step is performed by LLM agents reading the frozen Markdown directly; there is no deterministic Markdown parser or AST on the write side, and none is required. The deterministic proof obligation sits entirely on the reference grammar: every traceability link into the docs is a repository-relative line reference (`path:startLine-endLine`) that `semantic-extraction:check` resolves mechanically against the frozen manifest.

The semantic issue extraction pipeline may read `docs/architecture-map/architecture-map.yml` only to resolve and validate graph refs for generated issues. The map is not requirement source material. If the map and canonical docs disagree, canonical docs win and the map must be corrected.

The Source Map must bind every manifest file line to exactly one of three states:

```text
covered            -> at least one issue requirement line reference includes the line
explicitly_excluded -> an exclusion record names the line range and a reason class
auto_excluded      -> blank line, pure code-fence delimiter, or horizontal rule
```

A manifest line in none of these three states is a `semantic-extraction:check` failure: it means a documentation line was silently ignored.

### Requirement Extraction Rules

The extracting agents must treat a doc passage as potentially normative when it contains mandatory or boundary language such as:

```text
must
must not
required
forbidden
allowed only
never
source of truth
stable decision
contract
invariant
acceptance criteria
```

This word list is not the source of truth by itself. It is the minimum detection floor. The extracting agents must also catch normative tables, schemas, examples that define accepted behavior, and fenced contract blocks.

Normative statements embedded inside explanatory paragraphs must be extracted into atomic requirements instead of excluding the whole paragraph as supporting context. This is the rule that prevents an agent from skimming only the obvious feature sections and silently dropping security, isolation, cost, or error-handling rules buried in prose.

Every extracted requirement must include:

```text
id
title
statement
area
type
maturity
source line refs (path:startLine-endLine)
source excerpt hash
dependsOn
disposition
verificationLevel
openQuestions
```

The source excerpt hash is computed over the exact referenced lines, so a later baseline change to those lines is detectable.

`unknown` is not allowed for release-ready maturity, disposition, type, or verification level. If the extracting agents cannot classify a release-scoped requirement, `semantic-extraction:check` must fail.

Requirement IDs must be preserved across re-extractions by matching existing requirements in this order:

1. same requirement ID explicitly present in the previous catalog and still mapped to the same source lines;
2. same normalized statement with the same source line refs;
3. split, merge, amended, or removed relationship recorded during requirement change management.

IDs must not be reused after deletion. Removed requirements remain in the catalog with `removed` maturity or disposition and a removal reason.

## Traceability Coverage Report

`semantic-extraction:check` produces the traceability coverage report. The non-blank line rule for `total_doc_lines`: count every line of every active-frozen-manifest `files[]` document except mechanically auto-excluded lines (blank lines, pure code-fence delimiter lines, horizontal rules). The report must record, at minimum:

```text
baseline identity fields (baseline_id, baseline_version, manifest_path, manifest_hash, document_set_hash)
total_doc_lines
covered_doc_lines
explicitly_excluded_doc_lines
auto_excluded_doc_lines
coverage_percent
requirement counts by maturity and disposition
requirement counts by traceability status
list of current-scope requirements not yet verified
list of requirements covered only by a lower E2E level than they ultimately require
```

The report is regenerated on every extraction run and after every accepted Change Request. It must never be hand-edited to claim coverage the Source Map does not prove.

## Method Rehearsal

Before product implementation, the method itself must be rehearsed end-to-end on a small, throwaway example project so that a failure of the method is caught before it touches real product scope. The rehearsal exercises the full chain — freeze a tiny baseline, extract requirements, generate issues and the issue index, run `semantic-extraction:check` and `traceability:check`, run the fidelity review to an `iso` verdict, and drive one issue through the gate-first loop to a green gate — and commits a pass artifact at:

```text
spec/development/reports/method-rehearsal-report.md
```

`factory/dev-rehearsal.mjs` drives the rehearsal. The rehearsal is a blocking gate: no product implementation issue may start until the committed pass artifact records a passed verdict. A pass is proven by the artifact, never by assertion.

## Non-Negotiable Summary

- Freeze and verify the baseline before extraction.
- Cover every frozen doc line with an issue requirement reference or an explicit exclusion.
- Give every requirement a disposition; never `unknown` in a release-ready catalog.
- Author every issue from the frozen baseline through the intent-first template, gate-first.
- Keep one extraction write path and one deterministic check path.
