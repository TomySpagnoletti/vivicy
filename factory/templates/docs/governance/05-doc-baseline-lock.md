# 08 - Documentation Baseline Lock

Document status: stable development governance.

## Purpose

This document defines how {{PROJECT_NAME}} canonical product documentation becomes a cryptographically locked implementation baseline.

The goal is to make "frozen documentation" measurable. A development agent must be able to prove exactly which documentation version it is implementing, detect any unreviewed documentation drift, and route post-freeze changes through Product Change Requests instead of silently changing the target.

## Stable Decision

{{PROJECT_NAME}} uses a Canonical Documentation Baseline Lock before requirement extraction and implementation.

The lock is a machine-readable manifest generated by the repository-owned baseline tool:

```text
factory/doc-baseline.mjs
```

There is one baseline-lock mechanism. Do not create another hash script, external spreadsheet, manual checklist, or alternate baseline registry.

The baseline manifest stores:

- `baseline_id`;
- semantic `version`;
- `status`;
- repository git evidence;
- included and excluded paths;
- SHA-256 for every included source file;
- a deterministic `document_set_hash`;
- a deterministic `manifest_hash`;
- Change Request policy fields.

The manifest is the lock. The canonical docs are not made read-only with filesystem permissions. The development gate verifies hashes before extraction, issue generation, traceability updates, and implementation checkpoints.

## Baseline Statuses

Allowed statuses:

```text
draft
frozen
superseded
```

Status meanings:

- `draft`: generated for review, testing the mechanism, or comparing documentation states before owner freeze. It must not drive implementation.
- `frozen`: owner-approved source for requirement extraction and implementation. "Owner" here is the governance owner defined in [Source Of Truth](01-source-of-truth.md), not a product end user.
- `superseded`: replaced by a newer frozen baseline after accepted Change Requests or an explicit re-freeze.

Only a `frozen` baseline can be used as execution truth.

### Supersession Mechanics

The `superseded` state is reached by one mechanism only: when the baseline tool generates a new `frozen` manifest, it appends an unhashed `superseded` evidence block (`by_baseline_id`, `at`) to every prior `frozen` manifest in `docs/baselines/`. Like `generated_at` and the `git` block, the supersession marker is outside `manifest_hash`, so the prior manifest's hashed identity, recorded hashes, and file evidence are never edited — the old baseline stays fully auditable while it is mechanically retired.

A manifest carrying the supersession marker has effective status `superseded` regardless of its generation-time `status` field. `verify --require-status frozen` fails for it.

The active frozen baseline is therefore always the single `frozen` manifest without a supersession marker. Do not maintain a separate active-baseline pointer file or index.

### Freeze Preconditions

The tool refuses to generate a `frozen` manifest unless all of the following hold:

- git evidence is available and the working tree is clean, so the recorded `head_sha` is a committed state that reproduces the hashed bytes;
- owner-approval evidence is supplied with `--approved-by <approving-actor>` and `--approval-ref <recorded-approval-reference>`, recorded in the manifest `approval` block (`approved_by`, `approved_at`, `approval_ref`);
- `approval_ref` points at a durable record of the governance owner's freeze approval (for example a committed decision note, CR decision, or message reference), not a restatement invented by the agent.

A freeze performed from a dirty tree, an uncommitted canonical doc set, or without recorded owner approval is invalid. There is no dirty-freeze override flag.

## Versioning

Baseline versions use SemVer:

```text
MAJOR.MINOR.PATCH
```

Rules:

- `MAJOR`: stable architecture, source-of-truth hierarchy, isolation, protocol, security, or runtime boundary changed.
- `MINOR`: accepted current-build product behavior, feature scope, data contract, verification requirement, or implementation order changed.
- `PATCH`: wording, typo, reference, formatting, or clarification changed without changing expected behavior, architecture, scope, or verification.

The baseline ID format is:

```text
baseline-v<version>
```

Draft manifests may append `-draft`:

```text
baseline-v<version>-draft
```

The tool rejects baseline IDs outside this format and rejects a `-draft` suffix on a non-draft manifest. Manifests stored in `docs/baselines/` must be named `<baseline_id>.json`. The format is repository-agnostic — the product name is not repeated in the ID, since the manifest already lives in a project-named git repository.

Pre-1.0 rule: a `0.y.z` baseline is bootstrap-era. It may be frozen to exercise and govern the lock mechanism itself, but the first frozen baseline used for {{PROJECT_NAME}} product requirement extraction and implementation must be version `1.0.0` or higher, matching the `v1.0.0` example IDs in this document and in [Development Traceability Method](02-development-traceability-method.md). This deliberately overrides SemVer's "anything may change" reading of 0.x for the lock: a frozen 0.x manifest is still immutable, but it is not a valid implementation extraction source. From `1.0.0` on, the MAJOR/MINOR/PATCH rules above apply as written.

## Included Sources

The baseline tool includes only canonical product documentation:

- `docs/canonical/**/*.md`.

The tool excludes generated output, runtime dependencies, temporary folders, external repositories, support/governance documentation, evidence/spike documentation, the architecture map, and baseline manifests themselves.

Generated development-control artifacts under `spec/requirements/` and `spec/development/` are not included in the documentation baseline. They are derived from the frozen baseline and may change as extraction, issue generation, traceability, or local progress changes.

Actual Change Request instance files are controlled deltas, not part of the frozen baseline. The CR process documentation and template are governance/support files, not canonical product documentation.

Accepted CR text must not become an active requirement source outside the baseline. Accepted current-build CRs patch canonical docs first, then produce a new frozen baseline. Requirement extraction reads the patched canonical docs from that new baseline.

The architecture map is a derived visual/index artifact. It must cite only canonical docs, but it is not part of the canonical documentation baseline and must not become an independent product source.

Evidence/spike documents are not canonical product documentation. A spike can inform implementation or evidence verification, but any product decision discovered by a spike must be written back into `docs/canonical/` before it can become implementation truth.

After freeze, live implementation progress must not be recorded by changing `docs/architecture-map/architecture-map.yml`. The progress overlay is stored outside the baseline in `spec/development/issue-index.json` and `spec/development/progress-ledger.json`.

## Manifest Shape

The baseline manifest must be stable JSON. "Stable JSON" is a written serialization contract, not an emergent property of one script: recursively sorted object keys, `JSON.stringify` with 2-space indentation, UTF-8, LF line endings, one trailing newline, and hashed fields limited to strings, integers, booleans, arrays, and objects of those. Floats and locale-dependent values are forbidden in hashed fields. The contract is implemented by `stableJson` in `factory/doc-baseline.mjs`; any second consumer must reproduce it byte-for-byte rather than invent another canonical form.

Required top-level fields:

```json
{
  "schema_version": 1,
  "baseline_id": "baseline-v1.0.0",
  "version": "1.0.0",
  "status": "frozen",
  "product": "{{PROJECT_NAME}}",
  "generated_at": "2026-01-01T00:00:00.000Z",
  "generated_by": "factory/doc-baseline.mjs",
  "git": {
    "available": true,
    "head_sha": "<git-sha>",
    "working_tree_clean": true
  },
  "approval": {
    "approved_by": "<approving-actor>",
    "approved_at": "2026-01-01T00:00:00.000Z",
    "approval_ref": "<recorded-approval-reference>"
  },
  "change_request_policy": {
    "registry_path": "docs/change-requests",
    "accepted_current_build_required": true,
    "new_baseline_required_after_docs_change": true
  },
  "include": [],
  "exclude": [],
  "files": [],
  "document_set_hash": "<sha256>",
  "manifest_hash": "<sha256>"
}
```

`files` is sorted by path. Each file entry contains:

```json
{
  "path": "docs/canonical/01-<name>.md",
  "bytes": 12345,
  "sha256": "<sha256>"
}
```

`document_set_hash` is computed from the sorted file path, byte length, and file hash list. `manifest_hash` is computed from the manifest without the `manifest_hash`, `generated_at`, `git`, `approval`, and `superseded` fields.

`generated_at`, the `git` block, the `approval` block, and the `superseded` marker are evidence, not identity. They stay recorded in the manifest but are excluded from `manifest_hash`, so the hash is a pure function of the document set plus baseline metadata: regenerating an equivalent manifest for the same document set keeps the same manifest hash regardless of which commit, clock time, or approval record it was generated under, and a later supersession stamp does not invalidate the recorded hash.

The `approval` block is required for `frozen` manifests. The `superseded` block (`by_baseline_id`, `at`) appears only on manifests retired by a later freeze.

## Verification Rules

Before requirement extraction, the development agent must run:

```text
node factory/doc-baseline.mjs verify --manifest docs/baselines/<baseline>.json --require-status frozen
```

Verification must fail if:

- an included file hash changed;
- an included file was removed;
- a new included file appeared outside excluded paths;
- the manifest hash does not match;
- the manifest status is not allowed for the requested operation;
- the manifest carries a supersession marker but `frozen` was required;
- a `frozen` manifest records `working_tree_clean: false`, missing git evidence, or a missing/incomplete `approval` block;
- the manifest's `include`/`exclude` diverge from the repo-owned corpus policy, its `baseline_id` violates the governed format, or its filename does not match its `baseline_id`;
- the baseline ID or version does not match the implementation issue's declared canonical truth source.

The last check is pinned by the caller: development gates must pass `--require-baseline-id <baseline_id>` (and `--require-version <version>` when the issue declares it) so verification fails when the manifest's recorded identity is not the issue's declared truth source.

Implementation must not continue against a failed frozen baseline verification.

## Relationship To Requirement Extraction

Every generated traceability artifact must store:

```text
baseline_id
baseline_version
manifest_path
manifest_hash
document_set_hash
```

This applies to:

- Requirement Catalog;
- Source Map;
- Traceability Matrix;
- traceability coverage report;
- vertical implementation issues;
- Architecture Graph issue index;
- initialized progress-ledger baseline snapshot;
- verification gate registry.

Every requirement source reference must point to a source block whose parent file hash belongs to the active frozen canonical documentation baseline. Accepted current-build Change Requests must patch canonical docs and produce a newer frozen baseline before extraction uses them.

## Relationship To Change Requests

After a `frozen` baseline exists, product documentation is not patched directly from a raw idea.

The flow is:

```text
frozen baseline vX.Y.Z
  -> owner idea
  -> Change Request draft
  -> owner decision
  -> accepted_current_build
  -> canonical docs patched
  -> baseline tool generated and verified (prior frozen manifest stamped superseded)
  -> new frozen baseline vX.Y+1.0 or vX.Y.Z+1
  -> CR marked docs_applied
  -> traceability regenerated against new baseline
```

Accepted current-build CRs must record:

- previous `baseline_id`;
- previous baseline `version`;
- previous baseline `manifest_path`;
- previous `document_set_hash`;
- previous `manifest_hash`;
- target `baseline_bump`;
- resulting `baseline_id` after implementation;
- resulting baseline `version`;
- resulting baseline `manifest_path`;
- resulting `document_set_hash`;
- resulting `manifest_hash` after implementation.

The original baseline remains auditable. Its hashed identity — status at generation, file list, file hashes, `document_set_hash`, `manifest_hash` — is never edited retroactively. The only permitted later write is the unhashed `superseded` evidence block appended by the tool at the next freeze (see Supersession Mechanics).

## Implementation Gates

The baseline lock gate is required before:

- requirement extraction;
- source-map generation;
- traceability matrix generation;
- vertical issue generation;
- starting any implementation issue;
- marking an accepted current-build CR implemented;
- claiming documentation is frozen.

The gate command must be recorded in validation evidence.

## Non-Goals

This mechanism does not:

- replace Product Change Requests;
- replace Git history;
- replace requirement IDs or source block hashes;
- lock executable source code;
- make files read-only on disk;
- allow implementation from a draft baseline;
- create a second product specification.
