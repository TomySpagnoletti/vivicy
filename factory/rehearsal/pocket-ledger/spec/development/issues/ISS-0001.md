# ISS-0001 - Add an expense to the Ledger with field validation

## Summary

Implement the Ledger's add operation: it accepts an expense record, validates every field, rejects duplicate ids and malformed values, and stores accepted records in insertion order. The Ledger is the single source of truth for stored expenses, so no consumer module may mutate the store directly.

## Scope

Build the in-memory Ledger store and its `addExpense(record)` operation as a pure-JS module under `src/`.

An expense record has exactly these fields: `id` (a unique, non-empty string), `amountCents` (a positive integer amount in minor units / cents), `description` (a non-empty human-readable string), `date` (an ISO-8601 calendar date string in `YYYY-MM-DD` form), and `category` (a string category label assigned by the Categorizer).

`addExpense` stores accepted records in insertion order. It must reject, with a thrown error:

- an `id` that duplicates an already-stored record (ids are unique within a Ledger);
- an `amountCents` that is zero, negative, or non-integer;
- a `date` that does not match the `YYYY-MM-DD` form;
- a blank `description` or blank `id`.

The Ledger is the single source of truth for stored expenses: no other module may mutate the store directly, and consumers must not hold their own copy.

## Out Of Scope

Retrieval reads (ISS-0002), total computation (ISS-0003), categorization (ISS-0004), reporting, export, and CLI wiring. This issue does not assign categories; it stores whatever `category` value the caller supplies.

## Verification

Unit tests in `test/` (node:test) covering `addExpense` to full branch coverage of the governed code: a happy-path add that stores in insertion order; rejection of a duplicate id; rejection of zero, negative, and non-integer `amountCents`; rejection of a malformed `date`; rejection of a blank `description` and a blank `id`; and confirmation that stored records preserve insertion order across multiple adds. Each test asserts real behavior (returned/stored state or the thrown error), not merely that a function ran. The deterministic gate `gate:test:ledger-add-expense` must be green (node --test exit 0) before this issue is reported complete.
